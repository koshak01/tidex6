# ADR-001: Commitment scheme — `Poseidon(secret, nullifier)` only

**Status:** Accepted
**Date:** 2026-04-09

## Context

A privacy-preserving shielded pool needs a commitment scheme that hides the deposit data while still allowing the depositor to later prove ownership and spend the deposit. The commitment is what gets stored in the Merkle tree and is the public anchor that links a deposit to its eventual withdrawal — without revealing the linkage to outside observers.

Two questions need answers:

1. What goes inside the commitment hash?
2. Where do the auxiliary fields (auditor tag for selective disclosure, encrypted memo) live?

The MVP uses **fixed denominations** (0.1 / 1 / 10 SOL) — see ADR-007 indirectly. This is critical to the answer below: with fixed denominations the amount is publicly known by the program (it sees the lamports transferred) and does not need to be proven inside the ZK circuit.

## Decision

The commitment is the Poseidon hash of two secret values only:

```
commitment = Poseidon(secret, nullifier)
```

- `secret` — 32 random bytes generated by the depositor
- `nullifier` — 32 random bytes generated by the depositor

The auditor tag and encrypted memo, when present, are stored as **separate fields** in the `DepositEvent` struct, not inside the commitment:

```rust
pub struct DepositEvent {
    pub commitment: [u8; 32],          // Poseidon(secret, nullifier) — goes into Merkle tree
    pub root: [u8; 32],
    pub leaf_index: u64,
    pub timestamp: i64,
    pub auditor_tag: Option<Vec<u8>>,  // ElGamal ciphertext, NOT in commitment
    pub encrypted_memo: Option<Vec<u8>>,
    pub memo_ephemeral_pk: Option<[u8; 32]>,
}
```

## Consequences

**Positive:**
- The ZK circuit is minimal: prove knowledge of `(secret, nullifier)` such that `Poseidon(secret, nullifier)` equals a leaf in the Merkle tree, and that `Poseidon(nullifier)` equals the public `nullifier_hash`. Few constraints, low CU cost on verification, small attack surface.
- The privacy layer (Merkle tree + nullifiers) is fully decoupled from the disclosure layer (auditor tag + memo). A bug in the disclosure layer cannot compromise the privacy of users who do not opt into disclosure.
- Auditor tag and memo can evolve independently of the circuit. Adding a new memo encoding does not require a new trusted setup.

**Negative:**
- This commitment scheme is **only valid for fixed denominations**. When v0.3 introduces variable denominations, a new commitment that includes a Pedersen amount commitment + range proof will be required, along with a new circuit and a new trusted setup ceremony.
- The auditor tag is not bound to the commitment cryptographically. A malicious indexer or relayer could in principle drop the tag from the event log without affecting the validity of the deposit. Mitigation: integrators verify the event log directly via RPC, not via a third-party indexer they do not trust.

**Neutral:**
- The amount is publicly visible at the program level (the program sees the lamports transferred), so observers can categorize deposits by denomination. This is the same property Tornado-style fixed-denomination pools have always had, and it is what enables the fixed-anonymity-set model in the first place.
