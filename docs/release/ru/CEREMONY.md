# Церемония доверенной генерации параметров tidex6

> *Я даю доступ, а не разрешение.*

Публичная многосторонняя Phase-2 церемония, генерирующая production-параметры
Groth16 для `WithdrawCircuit<20>` — чтобы ни один человек в одиночку не мог
подделать доказательство. Работает на **<https://ceremony.tidex6.com>**.
Архитектурное решение: [ADR-017](../adr/ADR-017-public-ceremony-finalization.md).

## Зачем церемония

Groth16 требует Common Reference String, генерируемую один раз. При генерации
возникает секретная случайность — «токсичные отходы» (*toxic waste*). Тот, кто
её знает, может подделывать доказательства: выводить деньги, которые никогда не
вносились. Церемония делает setup многосторонним: каждый участник подмешивает
свою свежую случайность и уничтожает её. Итоговые параметры безопасны, пока
**хотя бы один** участник был честен — модель доверия 1-из-N. Каждый
дополнительный участник строго усиливает гарантию и никогда не ослабляет её.

Текущий on-chain ключ верификации — всё ещё однопользовательский development-
setup (помечен **DEVELOPMENT ONLY** в [`security.md`](../security.md) §1.4).
Церемония заменяет его.

## Как внести вклад

Откройте <https://ceremony.tidex6.com>, подключите кошелёк, нажмите
**Contribute**. Всё происходит в вашем браузере:

1. Свежая энтропия генерируется локально (`crypto.getRandomValues`).
2. Наш Rust-прувер (WebAssembly, `tidex6-prover-wasm::ceremony_contribute`)
   подмешивает её в текущие параметры. Случайность **не покидает вкладку** —
   на сервер уходят только новые параметры.
3. Координатор выполняет полную MPC-проверку (`mpc::verify_extension`):
   proof of knowledge вашего вклада, непрерывность delta-цепочки и
   неизменность остальных секций setup. Только после этого вклад принимается.

Один вклад на кошелёк. Полный круг занимает меньше минуты.

## Проверяй — не доверяй

Лог координатора — **не** источник истины; источник истины — опубликованный
транскрипт. Любой может скачать цепочку и перепроверить её независимо:

```text
curl -O https://ceremony.tidex6.com/transcript/genesis.state
curl -O https://ceremony.tidex6.com/transcript/current.state
git clone https://github.com/koshak01/tidex6 && cd tidex6
cargo run --release -p tidex6-circuits --bin ceremony_verify -- \
    ../genesis.state ../current.state
```

`ceremony_verify` перепроверяет, не доверяя серверу ни в чём:

- genesis-параметры внутренне согласованы (хеш схемы сходится);
- скачанное состояние принадлежит той же схеме;
- каждый вклад несёт валидный proof of knowledge, а delta-цепочка связывает
  genesis → … → current без выброшенных, переставленных или подменённых
  вкладов;
- печатает аттестацию (сжатую `delta_after`) каждого вклада — тот же hex, что
  показан на странице церемонии. Найдите свой кошелёк в выводе — и вы
  независимо доказали, что ваша случайность вошла в параметры.

## Финализация: drand-бикон

Церемонию закрывает **публичный бикон случайности**, а не человек:

1. Будущий раунд [drand](https://drand.love) `R` объявляется заранее. Его
   значение неизвестно никому, пока открыт приём вкладов, поэтому ни один
   участник — даже последний — не может сместить итоговые параметры.
2. Когда раунд `R` публикуется, его случайность применяется как финальный,
   *детерминированный* вклад:

   ```text
   cargo run --release -p tidex6-circuits --bin ceremony_finalize -- <round> <randomness_hex>
   ```

   Seed — `SHA-512("tidex6-ceremony-final-v1" ‖ round ‖ randomness)`, питающий
   ChaCha20 CSPRNG через **тот же проверенный путь `contribute`**, что и каждый
   человеческий вклад — никакого нового доверенного примитива. Любой может
   выполнить эту команду на опубликованном состоянии и воспроизвести финальные
   параметры бит-в-бит.
3. Состояние замораживается (`final.state` + аудит-запись `final.json`);
   дальнейшие вклады не принимаются.

## Извлечение VK и новый верификатор

On-chain ключ верификации извлекается из замороженного состояния:

```text
cargo run --release -p tidex6-circuits --bin ceremony_extract_vk -- final.state
```

Инструмент самопроверяет proving key (доказывает и верифицирует реальный
withdraw), затем перезаписывает `programs/tidex6-verifier/src/withdraw_vk.rs`.
Вывод **байт-детерминирован** от замороженного состояния — скачайте
`final.state`, запустите извлечение и сделайте `diff` с задеплоенной
программой.

По [ADR-005](../adr/ADR-005-verifier-non-upgradeable.md) текущий верификатор
неизменяем и не может быть пропатчен. Деплоится **новый** верификатор с
церемониальным VK, проходит source-верификацию, его upgrade authority
отзывается — неизменяемый, ровно как предшественник. Пулы мигрируют на него;
верификатор с dev-VK остаётся историческим.

## Сводка инструментов

| Инструмент | Роль |
|---|---|
| `ceremony_bootstrap` | одноразовая инициализация координатора: genesis → `~/.tidex6-ceremony/` |
| `tidex6-prover-wasm::ceremony_contribute` | вклад в браузере (энтропия остаётся во вкладке) |
| `ceremony_verify` | независимая публичная проверка всей цепочки |
| `ceremony_finalize` | детерминированная финализация drand-биконом + заморозка |
| `ceremony_extract_vk` | замороженное состояние → on-chain VK (байт-воспроизводимо) |

Всё это Rust (arkworks 0.5) — браузерный вклад, проверка координатора,
финализация и извлечение используют одну кодовую базу.
